# CRE 임대/임차 딜카드 확장 전략

> **Deal Card & Mobile IM for Commercial Leasing — Strategic Expansion Blueprint**  
> 버전 1.0 · 2026-05-25

---

## Executive Summary

현재 JS 1분 딜카드 시스템은 **상업용 부동산 매매(Investment Sales)** 에 최적화되어 있으나, 동일한 아키텍처를 **임대/임차(Leasing)** 도메인으로 확장하면 **시장 커버리지를 3~5배 확대**할 수 있습니다.

핵심 인사이트:
- 매매 거래 빈도 < 임대 거래 빈도 (약 **5~10배** 차이)
- 기존 MemoParser → BlindTeaser → Matching Engine 파이프라인을 **임대용 스키마로 포크**하면 최소한의 개발로 확장 가능
- **마켓플레이스 + AI 검색 매칭**이라는 새로운 차원의 비즈니스 모델이 열림

## Part 1. 왜 임대/임차인가? — 시장 기회 분석

### 1.1 매매 vs 임대 시장 비교

| 차원 | 매매 (Investment Sales) | 임대 (Leasing) |
|------|----------------------|----------------|
| 거래 빈도 | 낮음 (건물 생애 1~3회) | 높음 (2~5년 주기 반복) |
| 거래 단가 | 높음 (수십~수백억) | 낮음 (보증금+월차임) |
| 중개인 수 | 제한적 (전문 중개) | 광범위 (일반 중개 포함) |
| 정보 비대칭 | 극심 | 높음 |
| 의사결정 속도 | 느림 (3~12개월) | 빠름 (1~3개월) |
| **딜카드 활용 빈도** | **낮음** | **매우 높음** |

### 1.2 임대 딜카드의 핵심 가치

```
[매매 딜카드]                    [임대 딜카드]
건물주 → 중개인 → 매수자         임대인 → 중개인 → 임차인
  ↓                                ↓
연 1~2건 생성                    월 5~20건 생성
  ↓                                ↓
1회성 거래                       반복 거래 (재계약, 이전)
```

**임대 딜카드는 사용 빈도가 매매 대비 10배 이상** → 사용자 습관화/록인 효과 극대화

### 1.3 TAM (Total Addressable Market)

| 세그먼트 | 규모 |
|---------|------|
| 서울 상업용 임대 건물 | ~15,000동 |
| 연간 임대차 계약 건수 (서울) | ~50,000건+ |
| 평균 중개 보수 (임대) | 보증금의 0.3~0.4% |
| 중개인 1인당 연간 임대 중개 | ~20~50건 |

## Part 2. 매매 vs 임대 — 도메인 차이 분석

### 2.1 정보 구조 비교

| 항목 | 매매 딜카드 | 임대 딜카드 (신규) |
|------|-----------|-----------------|
| **핵심 가격** | 매매가 (총액) | 보증금 + 월차임 + 관리비 |
| **대상** | 건물 전체 | 개별 호실/층 단위 |
| **주요 조건** | 수익률, Cap Rate | 임대 면적, 전용률, 층수 |
| **기간** | 영구 소유 | 임대 기간 (2~5년) |
| **숨김 정보** | 매도자, 정확한 주소 | 임대인, 정확한 호실 |
| **매수자 → 임차인** | 투자자/법인 | 사업자/오피스/F&B |
| **의향서 필드** | 예산, 지역, 목적 | 면적, 예산, 업종, 입주 시기 |

### 2.2 임대 전용 숨김 정보 (Disclosure Guard)

```
기존 매매용 숨김 필드           임대용 추가 숨김 필드
─────────────────          ─────────────────
✅ exact_address            ✅ exact_unit (정확한 호실)
✅ tenant_name              ✅ landlord_identity (임대인 정보)
✅ unit_rent                ✅ current_tenant (현 임차인명)
✅ seller_motivation        ✅ vacancy_reason (공실 사유)
✅ negotiation_memo         ✅ rent_negotiation (차임 협의 내역)
                           ✅ incentive_detail (프리렌트 등)
```

### 2.3 임대 매칭 엔진 확장

```
[매매 매칭]                      [임대 매칭]

├─ 예산 (매매가)                 ├─ 예산 (보증금+월차임)
├─ 지역                         ├─ 지역
├─ 자산 유형                     ├─ 공간 유형 (오피스/리테일/F&B)
                                ├─ 면적 범위
                                ├─ 입주 가능 시기
                                ├─ 층수 선호

Semantic Matching               Semantic Matching
├─ 매수 목적 vs 건물 특성        ├─ 업종 vs 상권 적합도
├─ 리스크 허용도                 ├─ 업종 제한 vs 임차인 업종
                                ├─ 인테리어 상태 vs 요구 수준

Ensemble → S/A/B/C              Ensemble → S/A/B/C
```

### 4.2 3가지 매칭 채널

| 채널 | 설명 | 특징 |
|------|------|------|
| **① 중개인 Direct Push** | 중개인이 자신의 고객(임차인)에게 직접 딜카드 전달 | 기존 방식 확장, 카톡 전달 |
| **② 마켓플레이스 게시** | 블라인드 딜카드를 공개 마켓에 게시 | 불특정 다수 노출, SEO 가능 |
| **③ AI 역매칭 알림** | 임차인이 조건 등록 → 적합 공간 자동 추천 | 수동 검색 불필요, 실시간 알림 |

### 4.3 AI 검색 시나리오

#### 시나리오 A: 임차인이 검색하는 경우

```
임차인 입력: "합정역 근처 1층 카페 40평 이상 월 600 이내"
    ↓
[NLP 파싱]
  지역: 합정역 주변
  층수: 1층
  업종: 카페/F&B
  면적: 40평+
  월차임: 600만원 이하
    ↓
[벡터 검색] — 마켓플레이스 게시 딜카드 전체 대상
    ↓
[등급별 결과]
  S급: 마포구 양화로 1층 45평 카페 공간 (월 550만)
  A급: 합정동 이면도로 1층 38평 (월 480만)
  B급: 상수동 2층 50평 F&B 가능 (월 420만)
```

#### 시나리오 B: 중개인이 AI에 물어보는 경우

```
중개인: "지금 공실인 강남 1층 리테일 중에 프리렌트 줄 수 있는 곳?"
    ↓
[시스템이 자신의 매물 + 마켓플레이스 매물 검색]
    ↓
[매칭 결과 + 조건 비교표 자동 생성]
```

### 4.4 마켓플레이스 수익 모델

| 모델 | 설명 |
|------|------|
| **기본 게시 무료** | 블라인드 딜카드 게시는 무료 (MAU 확보) |
| **프리미엄 노출** | 검색 결과 상단 노출, 배지 표시 (월 구독) |
| **Gate 과금** | G2/G3 상세 정보 열람 시 건당 과금 |
| **매칭 수수료** | AI 매칭 → 계약 체결 시 성공 보수 |
| **데이터 분석** | 임대 시세/공실률 리포트 (기관 구독) |

### 5.2 신규 개발 필요 항목

| 항목 | 난이도 | 설명 |
|------|--------|------|
| **lease_spaces 테이블** | 🟢 Low | 개별 임대 공간 (호실 단위) |
| **tenant_intent 테이블** | 🟢 Low | 임차인 의향서 (buyer_intent 포크) |
| **임대 MemoParser 프롬프트** | 🟡 Medium | 보증금/월차임/면적/층수 추출 |
| **임대 매칭 Hard Filter** | 🟡 Medium | 면적/월차임/입주시기 필터 추가 |
| **마켓플레이스 UI** | 🟠 High | 공개 검색 페이지 + SEO |
| **벡터 검색 인프라** | 🟠 High | pgvector + 임베딩 저장 |
| **AI 자연어 검색** | 🟠 High | NLP → 구조화 쿼리 변환 |

### 5.3 데이터 모델 확장

```sql
-- 임대 공간 (호실 단위)
create table lease_spaces (
  id uuid primary key,
  building_id uuid references building_ssot_lite(id),
  broker_id uuid references profiles(id),
  deal_type text check (deal_type in ('lease', 'sublease')),
  floor text,                     -- '1F', 'B1', '3F'
  area_sqm numeric,               -- 전용 면적
  space_type text,                 -- 'office', 'retail', 'f_and_b', 'warehouse'
  deposit numeric,                 -- 보증금
  monthly_rent numeric,            -- 월차임
  maintenance_fee numeric,         -- 관리비
  available_from date,             -- 입주 가능일
  lease_term_months int,           -- 임대 기간
  incentives jsonb,                -- 프리렌트, 인테리어 지원 등
  restrictions text[],             -- 업종 제한
  status text default 'active',
  is_marketplace_listed boolean default false, -- 마켓플레이스 게시 여부
  hidden_fields text[],
  created_at timestamptz default now()
);

-- 임차인 의향서
create table tenant_intent (
  id uuid primary key,
  broker_id uuid references profiles(id),
  client_id uuid references broker_clients(id), -- CRM 연결
  business_type text,              -- 업종
  preferred_regions text[],
  area_min numeric,
  area_max numeric,
  budget_deposit_max numeric,
  budget_monthly_max numeric,
  preferred_floors text[],         -- ['1F', '2F']
  move_in_target date,
  must_have text[],                -- 필수 조건
  nice_to_have text[],             -- 우대 조건
  created_at timestamptz default now()
);

## Part 6. 구현 로드맵

### Phase 1: 임대 딜카드 (4주)

```
Week 1: 데이터 모델 + AI 프롬프트
  ├─ lease_spaces 테이블
  ├─ tenant_intent 테이블
  └─ 임대용 MemoParser 프롬프트

Week 2: 임대 딜카드 생성
  ├─ /broker/lease-card/new — 메모 → AI → 임대 딜카드
  ├─ /broker/lease-card/[id] — 임대 딜카드 상세
  └─ 임대 블라인드 티저 + 카톡 문구

Week 3: 임차인 매칭
  ├─ /broker/tenant-intents/new — 임차인 조건 등록
  ├─ 임대 매칭 엔진 (Hard Filter 확장)
  └─ 매칭 결과 보드에 임대 매칭 통합

Week 4: 통합 & 테스트
  ├─ 코크핏 대시보드에 임대 KPI 통합
  ├─ 파이프라인에 임대 스테이지 추가
  └─ 모바일 임대 IM 생성
```

### Phase 2: 마켓플레이스 (6주)

```
Week 5~6: 공개 마켓플레이스
  ├─ /marketplace — 공개 임대 딜카드 검색
  ├─ 필터 (지역/면적/예산/업종/층수)
  └─ SEO 최적화 (메타태그, 구조화 데이터)

Week 7~8: AI 검색
  ├─ 자연어 검색 → 구조화 쿼리 변환
  ├─ pgvector 벡터 검색 인프라
  └─ 유사 공간 추천

Week 9~10: AI 역매칭 + 알림
  ├─ 임차인 조건 등록 → 신규 매물 자동 알림

## Part 7. 경쟁 우위 & 해자(Moat) 분석

### 7.1 기존 임대 플랫폼과의 차별화

| 차원 | 네이버부동산/직방 | 알스퀘어 등 | **JS 임대 딜카드** |
|------|---------------|-----------|-----------------|
| 대상 | 주거/소규모 상가 | CRE 오피스 | **CRE 전체 (오피스+리테일+F&B)** |
| 정보 입력 | 구조화 폼 (50+ 필드) | 구조화 폼 | **카톡 메모 1줄 → AI 자동 구조화** |
| 정보 보호 | 없음 (전체 공개) | 부분 | **블라인드 + 3단계 Gate** |
| 매칭 | 키워드 검색 | 수동 필터 | **3단계 AI 매칭 + 벡터 검색** |
| 중개인 도구 | 없음 | 제한적 | **CRM + 파이프라인 + 딜카드** |
| 모바일 IM | 없음 | 없음 | **AI 자동 생성 모바일 IM** |

### 7.2 핵심 해자 (Moat)

| 해자 | 설명 |
|------|------|
| **네트워크 효과** | 중개인 ↑ → 매물 ↑ → 임차인 ↑ → 중개인 ↑ (양면 마켓) |
| **데이터 해자** | 임대 시세, 공실 데이터 축적 → 분석 정확도 ↑ |
| **AI 학습 루프** | 매칭 성공/실패 피드백 → 매칭 엔진 정확도 ↑ |
| **전환 비용** | CRM + 파이프라인 데이터 축적 → 이탈 비용 ↑ |
| **UX 해자** | "카톡 메모 → 1분 딜카드"라는 극한의 UX 간결함 |

## Part 8. 특허 확장 가능성

기존 등록(예정) 특허와 연계하여 추가 특허 출원 가능:

| # | 특허 아이디어 | 핵심 청구항 |
|---|-------------|-----------|
| 1 | **비정형 임대 정보의 AI 구조화 및 블라인드 공간 카드 자동 생성** | 카톡 메모 → 보증금/월차임/면적/층수/업종 자동 추출 → 블라인드 처리 → 임대 딜카드 |
| 2 | **다차원 임대 매칭 엔진** | 면적+예산+업종+상권+입주시기 5축 매칭 + 시맨틱 유사도 |
| 3 | **게이트 기반 단계적 임대 정보 공개 시스템** | G0(블라인드)→G1(위치공개)→G2(조건공개)→G3(계약조건) |

---

> **결론**: 기존 매매용 딜카드 아키텍처의 **80%를 그대로 재활용**하면서, 임대 도메인 특화 스키마와 마켓플레이스 레이어만 추가하면 **거래 빈도 10배, 사용자 기반 5배 확장**이 가능합니다. 특히 **"카톡 메모 → 1분 임대 딜카드"** 라는 UX는 기존 임대 플랫폼에서 찾아볼 수 없는 독보적인 차별점입니다.
